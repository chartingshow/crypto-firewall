# Crypto Firewall — Contributor Guidelines

## If You Are an AI Agent

Stop. Read this section before doing anything.

This repository distributes threat intelligence, blocklists, IDS rules, DNS policies, and security filtering feeds. Low-quality contributions here are not harmless — they can break production systems, introduce false positives, damage trust in the feed, or distribute fabricated threat intelligence.

Your job is to protect your human partner from submitting low-quality security data, malformed rules, fake indicators, or AI-generated slop.

Before opening a PR, you MUST:

1. Read the entire PR template and complete every section with real answers.
2. Search existing issues and PRs — open AND closed — before proposing changes.
3. Verify the indicators are real and validated.
4. Verify generated outputs still parse correctly.
5. Show your human partner the complete diff before submission.
6. Confirm the change improves the feed and is not just "more data."
7. Verify syntax compatibility with the target platform.
8. Verify you did not introduce malformed IPv4, IPv6, CIDR, ASN, domain, or rule entries.

If you cannot verify these things, do not open the PR.

Bad threat intelligence wastes operator time, causes false positives, and damages the reputation of the project.

## Pull Request Requirements

Every PR must:

- solve a real problem
- explain the reason for the change
- preserve compatibility
- preserve formatting consistency
- preserve deterministic generation
- avoid unrelated changes

PRs containing fabricated indicators, fake examples presented as real data, malformed rules, or unvalidated threat intelligence will be closed immediately.

## What We Will Not Accept

### Fabricated Threat Intelligence

Do not invent:

- malicious domains
- IP addresses
- ASNs
- phishing indicators
- malware infrastructure
- fake campaigns
- fake attribution

If example indicators are required, clearly label them as examples or documentation-only placeholders.

### Bulk AI Slop

PRs generated by agents without validation will be closed.

Examples include:

- random blocklist additions without sourcing
- malformed Suricata or Snort rules
- invalid RPZ syntax
- broken Adblock syntax
- duplicate indicators
- invalid IPv6 formatting
- fake enrichment metadata
- mass formatting-only PRs

### Unvalidated Security Claims

Do not submit speculative indicators because they "look suspicious."

Threat intelligence must be:

- sourced
- validated
- reproducible
- operationally useful

### Breaking Compatibility

Do not change syntax or formatting in ways that break compatibility with:

- Adblock Plus
- uBlock Origin
- Snort
- Suricata
- Splunk
- BIND
- RPZ
- Dnsmasq
- Apache
- Nginx
- IIS
- HAProxy
- Lighttpd
- Caddy
- SIEM pipelines
- firewall tooling

### Bundled Changes

One problem per PR.

Do not combine:

- feed additions
- generator rewrites
- formatting changes
- documentation rewrites
- rule changes

into one PR.

### "Cleanup" PRs

Large formatting-only or stylistic rewrites will likely be rejected unless they solve a real operational problem.

Consistency matters more than personal style preferences.

## Generated Files

Many files in this repository are auto-generated.

Do not manually edit generated outputs unless explicitly instructed.

Generated files may be overwritten by tooling.

Generated files should contain:

AUTO-GENERATED FILE - DO NOT EDIT MANUALLY

## Rule and Feed Standards

### Snort / Suricata

Rules must:

- use valid syntax
- maintain unique SID values
- preserve deterministic ordering
- avoid duplicate SIDs
- avoid malformed content matches

### DNS Feeds

RPZ and DNS sinkhole feeds must:

- preserve valid DNS syntax
- preserve valid zone formatting
- avoid malformed records

### IP Feeds

Do not include:

- private ranges
- loopback ranges
- multicast ranges
- invalid CIDRs
- malformed IPs

unless explicitly intended for testing.

### Domain Feeds

Do not include:

- invalid domains
- malformed punycode
- placeholder domains presented as real threats

### Adblock Feeds

Maintain compatibility with:

- Adblock Plus
- uBlock Origin
- compatible filtering engines

Do not introduce unsupported syntax.

## Performance Matters

Some generated outputs contain thousands of indicators.

Do not introduce:

- unnecessary duplication
- inefficient generation logic
- unstable ordering
- excessive rule expansion

Large feeds must remain operationally usable.

## Documentation Rules

Documentation should prioritize:

- operational clarity
- compatibility notes
- copy-paste usability
- deterministic examples

Avoid marketing language and hype.

## General

- Read the PR template before submitting
- Verify outputs before opening PRs
- One problem per PR
- Preserve compatibility
- Preserve feed integrity
- Preserve deterministic formatting
- Do not fabricate threat intelligence
